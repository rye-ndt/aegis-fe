# Relay Swap — Mini-App Plan

Backend-driven, one-step-at-a-time signing. The mini app already does this for
`SignRequest` with `autoSign: true`. The `/swap` flow **reuses** that path —
no new `RequestType`, no new handler in v1. The only mini-app change is a
small correctness fix so the window stays open until all steps in a swap
queue are signed.

Cross-reference: `be/constructions/relay-swap-plan.md` — this file covers
only the FE side.

## 1. Why not a new `RequestType`?

Relay returns N transactions per quote (typically 1 approval + 1 swap; more
for complex cross-chain routes). Our existing `SignRequest` already encodes
"here is one tx, sign it autonomously and report back." The backend will emit
**N `SignRequest`s in sequence** — each new one appearing in Redis only after
the previous `SignResponse` resolves.

From the mini-app's point of view there is no "swap" concept. It just gets a
`sign` request, signs, posts a response, and closes… which is the problem
that §2 addresses.

Introducing a `SwapRequest` that carries the full step list would be cleaner
UX (single open/close cycle) but:
- Duplicates `createSessionKeyClient` logic per handler type.
- Moves Relay-quote understanding into the FE, which the user explicitly
  rejected (backend forms the request body, backend talks to Relay).

Deferred to v2. Flagged in `be/constructions/relay-swap-plan.md §2`.

## 2. The one real FE change

**Problem.** `App.tsx` dispatches on `request.requestType`. On
`SignResponse` post, `SignHandler` calls `Telegram.WebApp.close()` after ~1.5s.
For a 2-step swap, that closes the window before the second `SignRequest` is
emitted — the user sees tx 1 land, window closes, tx 2 never signed.

**Fix — minimal, no new types.** After a successful `autoSign` response,
**before** closing, re-fetch `/request/:requestId?next=true` (new query flag,
see backend §2 below). If the backend returns a fresh pending request for the
same user, stay open and dispatch it; only close when `404`.

Concretely:

### Files modified

| Path | Change |
| ---- | ------ |
| `src/components/handlers/SignHandler.tsx` | After `reportTxHash` resolves successfully, call a new hook `useNextSwapStep({ previousRequestId, userId })`. If it returns a new `SignRequest`, swap `request` state and re-run the auto-sign effect. Else, `Telegram.WebApp.close()` as today. |
| `src/hooks/useRequest.ts` | Accept an optional `{ after: requestId }` arg that appends `?after=<id>` to the `/request/:id` URL. Backend uses this to return the next queued request in the same swap session (or `404`). |
| `src/App.tsx` | No change — dispatch is already request-type driven; the next request is also a `sign`, so the same handler stays mounted. |
| `src/components/handlers/SignHandler.tsx` | Add a small "Step X of Y" line to the auto-sign UI, sourced from `request.description` if backend formats it as `"Relay swap step N/M"` (it does — see backend §4). |

### Files added

None in v1.

## 3. Session-key path — unchanged

`createSessionKeyClient` in `utils/crypto.ts` already handles:
- Reading the encrypted blob from Telegram CloudStorage (`useDelegatedKey`).
- Building the ZeroDev Kernel client with the session signer.
- `sendTransaction({ chain: null })` with paymaster when configured.

Nothing in that file changes. Every step of a Relay quote is just another
`(to, data, value)` triple.

**Critical**: the Kernel client instance is cheap to reuse but costs ~200–400ms
to initialise (public client + kernel account resolution). If we dispatch
multiple steps in one session, **cache** the session client across steps
inside `SignHandler` (store the resolved client in a `useRef` keyed by the
encrypted blob string). Otherwise each step pays the full init cost. This is
an FE-only optimisation; start with re-init per step if it's simpler, profile
later.

## 4. Aegis-Guard re-approval interaction

If the backend's Aegis-Guard check fails before it even calls Relay, it pushes
an `ApproveRequest` (subtype `aegis_guard`) — not a `SignRequest`. Existing
`ApproveHandler` + `ApprovalOnboarding` handle it unchanged. After the user
re-approves, they have to run `/swap` again. No mini-app code change for this
path.

(If we later chain reapproval → swap in one mini-app session we'd need a
small state machine; out of v1 scope.)

## 5. Button-only UX — confirmation

User requirement: "one button to open the mini app, and that's it." With the
`autoSign: true` path already in place this is already the UX:

1. Telegram message arrives with an inline-keyboard button (generated by
   `artifactRenderer/telegram.ts` for `kind: 'sign_calldata'` artifacts).
2. User taps button → Telegram WebApp opens at `?requestId=…`.
3. `App.tsx` → `SignHandler` with `autoSign: true` → session key signs →
   tx hash posted → fetch-next-step loop → close when queue is empty.
4. No manual confirmation, no signature popups, no field entry.

## 6. Testing matrix

Smoke-test these paths in order:

1. Same-chain swap with no approval needed (e.g. native AVAX → USDC on Fuji).
   Expect: 1 SignRequest, 1 tx, close.
2. Same-chain swap needing ERC-20 approval (USDC → WAVAX). Expect: 2
   SignRequests, 2 txs, the window stays open across them.
3. Cross-chain swap (Base USDC → Arbitrum ETH). Expect: N SignRequests per
   Relay's route; all signed; final close.
4. Aegis-Guard short: cap = 10 USDC, user swaps 20 USDC. Expect: a single
   `ApproveRequest` (aegis_guard reapproval). No Relay calls. User re-
   approves, runs `/swap` again.
5. User rejects step 2 of 3 in the manual fallback modal. Expect: backend
   receives `rejected: true`, aborts queue, emits a chat error. Window closes.

## 7. Out of scope (v1)

- Purpose-built `SwapRequest` type.
- Quote preview inside the mini app (fees, minReceived, expected output).
  Backend sends the chat a summary instead.
- Mid-swap cancel button. (User can reject the next step in manual mode.)
- Destination-fill polling UI for cross-chain.

## 8. Status-log entry to write after implementation

When the feature ships, add to `fe/privy-auth/status.md`:

```
## /swap — 2026-04-24

SignHandler now supports multi-step swaps by fetching the next queued
request (if any) after a successful autoSign response, before calling
Telegram.WebApp.close(). New `?after=<previousRequestId>` query is passed to
GET /request/:id to express the "what's next for this user" semantics.

Convention introduced: a mini-app handler MAY keep the WebApp window open
across multiple same-type requests when the backend signals a continuation.
The default remains close-after-one. Only SignHandler participates today;
other handlers retain their close-on-success behaviour.
```
